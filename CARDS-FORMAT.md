# CARDS-FORMAT — 卡片内容生成规则

> 渲染 Step 3 HTML 时，每张卡需要 AI 现场生成 4 个字段。本文规定这些字段的写法。

---

## 输入数据

| 来源 | 用法 |
|------|------|
| `<SKILL_DIR>/cards/thinking-cards.json` | 12 张卡的 `id` / `name` / `tagline` / `prompt_pattern` / `good_for` / `stage` |
| `<SKILL_DIR>/identity/<id>.md` | 当前身份的视角与关键词 |
| Step 2 选中的素材 | 标题 / 论点 / 数据 / 链接 |
| `scripts/state.mjs get` 里的 `last_cards` | 避开昨天用过的卡 |

---

## 输出：3 张卡 × 4 字段

每张卡的 JSON 必须包含：

```json
{
  "id": "first-principles",
  "name": "🔬 第一性原理",
  "tagline": "剥到不能再剥",
  "logic": "...",
  "reading": "...",
  "questions": ["Q1...", "Q2...", "Q3..."]
}
```

### 字段 1 — `logic`（底层逻辑）

- ≤60 字
- 一句话讲清楚这个思维模型的核心机制
- 不要举例，不要类比
- 例：`"把现象拆到最基本的物理/经济/人性事实，再从这些事实重新推导，跳过行业惯例的中间假设。"`

### 字段 2 — `reading`（本事件解读）

- ≤120 字
- **必须带入当前身份**，用"作为 [身份]，..."或类似句式开头
- 用该思考模型对今日素材做 30 秒解读
- **要有锋利立场**，不写"可能"、"也许"、"值得关注"这类废话
- 例（身份=AI 解决方案专家，素材=Karpathy 加入 Anthropic）：
  > `"作为 AI 解决方案专家，这意味着 Anthropic 已经从'安全派研究院'切换到'要赢的产品公司'。Karpathy 不去做研究，他去做工程基建——这是 OpenAI 2023 年的剧本。下一步必有面向开发者的杀手级工具发布。"`

### 字段 3 — `questions`（三个追问）

- 数组长度恰好 3
- 每题 1-2 句话，自带立场对抗
- **层次递进**：
  1. **判断题**：基于事实让用户表态（例："你觉得这是真信号还是噪音？为什么？"）
  2. **反向/假设题**：拿一个反事实逼用户压力测试（例："如果反过来发生 X，你的判断会怎么变？"）
  3. **行动含义题**：把抽象判断落到用户自己的下一步（例："基于这个判断，你下周必须开始做的 1 件具体事是什么？"）

---

## 提问的硬规则（极其重要）

| 反例 ❌ | 正例 ✅ |
|---------|---------|
| "你怎么看这件事？" | "如果 OpenAI 明天倒闭，火山引擎的客户会更慌还是更稳？为什么？" |
| "你觉得 AGI 会怎么发展？" | "假设 2027 年 GPT-7 比 GPT-5 强 10 倍但贵 100 倍，谁会先死：模型公司还是应用公司？" |
| "你怎么应对这个趋势？" | "如果你只能在下季度做 1 件事来抓住这个机会，是什么？为什么不是其他事？" |

**通用配方**：
- 不允许"开放式空话"，必须有**两个选项的对抗**或**极端约束**
- 必须包含一个**具体对象**（人/公司/产品/数字），不是抽象概念
- 用户答完一题，AI 反馈 ≤3 句（但 HTML 模式下 AI 不参与作答，所以反馈逻辑只在 Step 4 沉淀时由 sediment.mjs 把用户原话原样保留）

## 抽卡规则（智能匹配，非随机）

从 `thinking-cards.json` 的 **12 张卡**里抽 3 张时，**按与今日素材的匹配度排序**，不是随机抽：

1. **判断事件类型** — 看 Step 2 选中的那条素材属于下方匹配表的哪一类（一条素材通常命中 1-2 行）
2. **按匹配表 + 卡片 `good_for` 字段，选出匹配度最高的 3 张**
3. **排除** `state.mjs get` 返回的 `last_cards` 数组（昨天用过的 3 张）；剔除后若剩 < 3 张，放宽此规则
4. 匹配度优先于多样性 —— 若匹配度最高的 3 张 `good_for` 有重叠，**保留匹配度**，不为凑多样性强行换卡
5. 用完登记：`node <SKILL_DIR>/scripts/state.mjs remember-cards <id1>,<id2>,<id3>`

### 事件类型 → 推荐模型 匹配表

| 事件类型 | 优先推荐的卡 |
|---------|------------|
| 厂商发布会 / 通稿 / 跑分刷榜 | 🧩 5W2H、📡 信号噪音、🧠 认知偏差 |
| 号称"范式转移"的技术突破 | 🔬 第一性原理、📈 Gartner曲线 |
| 完美 Demo / 高估值融资 | 🔄 逆向思维、⚖️ 供需框架 |
| 价格战 / 开源爆发 / 商业动作 | 🔗 价值链、🎲 第二层思考 |
| 全网同质化狂欢 or 集体悲观 | 🎲 第二层思考、🧠 认知偏差 |
| 趋势预测 / 周期定位 | 🌉 类比、📈 Gartner曲线 |
| 新证据与旧判断冲突 | 🎯 贝叶斯更新 |
| 要把洞察讲给客户/高管 | 🏛️ SCQA |
| 判断某 AI 方向能否成市场 | ⚖️ 供需框架、🔗 价值链 |

> 匹配理由不必单独成字段——在生成每张卡的 `reading`（本事件解读）时，让"为什么这张卡适合这条素材"自然体现出来即可。

## 常见错误自查

生成完 3 张卡 JSON 后，用户提交前自检：
- [ ] `reading` 里出现了身份关键词？（如"作为 [身份]"）
- [ ] `reading` 字数在 80-120 之间？
- [ ] 3 个问题里至少 2 个含有具体对象（人名 / 公司名 / 数字）？
- [ ] 3 个问题层次清晰（判断 → 反向 → 行动）？
- [ ] `last_cards` 已避开？
